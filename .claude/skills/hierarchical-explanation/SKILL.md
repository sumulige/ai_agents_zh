# 分层解释法 (Hierarchical Explanation)

> 让复杂概念变得易于理解和渐进式学习的写作技能

**版本**: 1.0.0
**作者**: sumulige
**标签**: [documentation, education, writing-style]
**难度**: 中级

---

## [Level 1] 技能本质

**分层解释法**是"知识阶梯"写作法，让读者按需获取信息深度。

一句话：像导游图一样，让读者选择自己的理解深度。

> 💡 **够用了吗？** 知道这是"渐进式披露"的写作方法就够了。继续阅读 [Level 2] 了解核心概念。

---

## [Level 2] 核心概念

### 三大原则

| 原则 | 说明 | 类比 |
|------|------|------|
| **渐进式披露** | 从简单到复杂，从本质到细节 | 像剥洋葱，一层层深入 |
| **读者自适应** | 不同角色需要不同深度 | 像地图，有全景有特写 |
| **出口点设计** | 每层提示读者是否继续 | 像游戏存档点，随时可停 |

### 适用场景

- 技术文档（README, API 文档）
- 教学材料（教程、课程）
- 项目文档（启动文档、任务计划）
- 任何复杂概念的讲解

> 💡 **了解够了？** 理解三大原则即可应用基本模式。继续阅读 [Level 3] 掌握五层结构。

## [Level 3] 五层结构

### 层级概览

```
┌─────────────────────────────────────────────────────────────┐
│                  分层解释金字塔                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    Level 5: ████████  进阶模式 (30分钟+)    │
│                  Level 4: ██████████  实现细节 (15分钟)      │
│                Level 3: ████████████  组成拆解 (5分钟)       │
│              Level 2: ███████████████  核心概念 (1分钟)      │
│            Level 1: ██████████████████  一语道破 (5秒)       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各层规范

| 层级 | 时间 | 目标 | 读者 | 标记 |
|------|------|------|------|------|
| **Level 1: Essence** | 5秒 | 快速定位 | 所有人 | `[Level 1]` |
| **Level 2: Concepts** | 1分钟 | 理解核心 | 项目成员 | `[Level 2]` |
| **Level 3: Components** | 5分钟 | 掌握结构 | 执行者 | `[Level 3]` |
| **Level 4: Details** | 15分钟 | 可执行 | 实施者 | `[Level 4]` |
| **Level 5: Advanced** | 30分钟+ | 专家级 | 专家者 | `[Level 5]` |

### 层级模板

#### Level 1: 一语道破
```markdown
## [Level 1] {TITLE}

**{CONCEPT}** 是 {ANALOGY}，旨在 {PURPOSE}。

一句话：{ONE_LINER}
```

#### Level 2: 核心概念
```markdown
## [Level 2] {TITLE}

本项目/概念围绕 {N} 个核心要素：

### 要素 1: {NAME}
- **定义**: {DEFINITION}
- **重要性**: {WHY_IT_MATTERS}
```

#### Level 3: 组成拆解
```markdown
## [Level 3] {TITLE}

### 架构图
```
{ASCII_ART}
```

### 组件说明
| 组件 | 职责 | 依赖 |
|------|------|------|
| {COMPONENT} | {RESPONSIBILITY} | {DEPENDENCY} |
```

#### Level 4: 实现细节
```markdown
## [Level 4] {TITLE}

### 步骤 1: {STEP_TITLE}
**目标**: {OBJECTIVE}
**操作**:
1. {ACTION_1}
2. {ACTION_2}

### 代码示例
```{LANG}
{CODE}
```
```

#### Level 5: 进阶模式
```markdown
## [Level 5] {TITLE}

### 模式 1: {PATTERN_NAME}
- **使用场景**: {WHEN_TO_USE}
- **实现方法**: {HOW_TO}
- **注意事项**: {CAVEATS}
```

## [Level 4] 应用场景与写作指南

### 技术文档
```
README.md
├── [Level 1] 这是什么项目？
├── [Level 2] 核心功能
├── [Level 3] 架构设计
├── [Level 4] 安装使用
└── [Level 5] 高级配置
```

### 教学材料
```
教程章节
├── [Level 1] 这一节学什么？
├── [Level 2] 关键概念
├── [Level 3] 原理图解
├── [Level 4] 动手实践
└── [Level 5] 扩展应用
```

### 项目文档
```
PROJECT_KICKOFF.md
├── [Level 1] 项目本质
├── [Level 2] 核心目标
├── [Level 3] 约束边界
├── [Level 4] 职责划分
└── [Level 5] 启动清单
```

### 写作指南

#### 原则 1: 每层独立可读
- ✅ 每层可以独立理解
- ✅ 不要说"如上文所述"
- ✅ 交叉引用用链接而非重复

#### 原则 2: 清晰的层级标记
- ✅ 使用 `[Level N]` 前缀
- ✅ 使用分隔符 `---` 区分层级
- ✅ 保持视觉一致性

#### 原则 3: 出口点设计
```markdown
> 💡 **够用了吗？** 如果你是 {ROLE}，可以在此止步。
> 需要更多信息？继续阅读 [Level N+1]。
```

#### 原则 4: 视觉化辅助
- ✅ 使用 ASCII 艺术图
- ✅ 使用表格展示结构
- ✅ 使用代码块展示示例

### 常见错误

| 错误 | 为什么不好 | 正确做法 |
|------|------------|----------|
| 每层重复相同内容 | 浪费读者时间 | 每层提供新信息 |
| 没有出口点提示 | 读者不知道何时停 | 每层结束加 💡 提示 |
| 层级跳跃太大 | 读者跟不上 | 保持渐进坡度 |
| 缺少视觉分隔 | 难以定位层级 | 用 `---` 和标记 |

> 💡 **需要精通？** 掌握写作规范后，继续阅读 [Level 5] 了解最佳实践。

## [Level 5] 最佳实践与示例

### 实践 1: 类比的力量

Level 1 的类比至关重要：

```
❌ n8n 是一个工作流自动化工具
✅ n8n 是可视化的乐高积木，用于搭建自动化流程

❌ Docker 是容器化平台
✅ Docker 是集装箱，让软件在任何船上都能运行
```

### 实践 2: 角色化出口点

针对不同角色设计出口点：

```markdown
> 💡 **够用了吗？**
> - 项目干系人：在此止步
> - 开发者：继续阅读 [Level 4]
> - 运维人员：跳转到 [Level 5] 部署章节
```

### 实践 3: ASCII 艺术图

简单的图表胜过千言万语：

```
数据流图:
[用户] → [API] → [服务] → [数据库]
   ↓        ↓        ↓         ↓
 错误    验证     处理      持久化
```

### 实践 4: 渐进式复杂度

确保每一层都比上一层略复杂：

```
Level 1: "n8n 是连接工具的 glue"
Level 2: "节点、连接、工作流三个概念"
Level 3: "触发器 → 处理 → 输出 的数据流"
Level 4: "安装 n8n，创建第一个工作流"
Level 5: "子工作流、错误处理、生产部署"
```

### 完整示例：解释 "n8n"

#### Level 1
```markdown
## [Level 1] 什么是 n8n？

**n8n** 是可视化的乐高积木，用于连接各种服务自动执行任务。

一句话：像搭积木一样搭建自动化流程，不用写代码。
```

#### Level 2
```markdown
## [Level 2] 核心概念

### 三个关键概念
- **节点(Node)**: 一块积木，代表一个操作（如发送邮件）
- **连接(Connection)**: 积木之间的粘连，决定数据如何流动
- **工作流(Workflow)**: 搭好的整体，完整的自动化流程
```

#### Level 3
```markdown
## [Level 3] 工作流结构

```
┌─────────────────────────────────────────────────────────────┐
│                    n8n 工作流结构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [触发器] ──► [处理节点1] ──► [处理节点2] ──► [输出]        │
│      │           │               │              │          │
│   定时/事件     转换数据        调用API        保存结果      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
```

#### Level 4
```markdown
## [Level 4] 快速开始

### 步骤 1: 安装 n8n
```bash
npm install n8n -g
n8n start
```

### 步骤 2: 创建工作流
1. 打开浏览器访问 http://localhost:5678
2. 点击 "Add workflow"
3. 拖入一个 "Cron" 节点作为触发器
...
```

#### Level 5
```markdown
## [Level 5] 进阶技巧

### 子工作流复用
将重复的流程提取为子工作流，在多个工作流中调用。

### 错误处理
使用 "Error Trigger" 节点捕获失败，发送通知。

### 生产部署
使用 Docker 或 n8n Cloud 部署，配置环境变量管理敏感信息。
```
